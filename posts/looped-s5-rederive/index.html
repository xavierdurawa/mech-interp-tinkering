Looped transformers solve S5 by re-deriving, not by carrying state

I hoped forcing recurrence would make a transformer carry a running state. It length-generalizes better — but a causal swap says it’s still re-deriving every step. And getting even that to happen reliably is its own puzzle.

What I was thinking

Beginning to work on a longer-term project to try and identify slices of a circuit that operate on spans larger than a single token. As a starting point, I wanted to study a simplified toy model capable of the cleanest version of that: tracking a running state across a sequence. Permutation composition on S₅ is a nice fit — you’re handed a sequence of permutations and have to report the running composition after each one — because it forces a genuine span-level computation and there’s a sharp distinction between two ways to do it.

You can carry a running state and update it each step (the “real” span-level algorithm — the thing I ultimately want to find circuits for), or you can re-derive the state at each position from scratch by looking back over all the raw actions (a shortcut that happens to work). Which one a model actually does is exactly the kind of question the broader project is about, and S₅ lets me ask it in a setting clean enough to answer causally.

The goal is a model that solves the task the way you’re supposed to: hold a running state and update it each step. That carried state is the span-level computation I want to study, so I need a model that genuinely builds one — not one that fakes the answer with a lookback shortcut.

Plain transformers can learn to do this — Li et al. show they’ll sometimes find the proper carry-style algorithm and length-generalize impressively far. The catch is that they find it unreliably: whether a given training run lands on the genuine algorithm or on a brittle shortcut that breaks past the training length seems to come down to the luck of initialization and data order.

My hope was that a looped transformer — a weight-tied block repeated more times for longer inputs, à la Fan et al. — would make the carry solution the default rather than a lucky draw, by handing the model depth that scales with the input the way a genuine step-by-step computation would need. But this turned out to just provide the model with the extra depth it needed to better execute the shortcut computational approach — re-deriving the state more effectively — without ever learning the carried state.

What I actually did (in order)

First, just try to reproduce Li. I trained a plain Pythia-160M on the S₅ task the way the paper describes and gated it for length generalization. It learned to track — but only shallowly, holding up to about depth 9–12 before falling apart, nowhere near the numbers the paper reports (up to ~106). So: a genuine tracker, but a weak one, and not the reliable long-range carry I was after.

Maybe it was the seed. Li is explicit that the good algorithm is a lottery of initialization and data order — they report proportions over as many as 20 replicates. So I ran a few more seeds. They all clustered in the same 9–12 band. No seed broke out into the deep-generalizing regime; the variation was in shallow tracking depth, not in whether it generalized at all.

Was the training even trending the right way? Before sinking a full multi-day sweep into seed-hunting, I ran a couple of truncated runs and looked at the early mechanism signature — was anything heading toward the good solution given more time? Nothing obviously was. The runs looked like they were settling into the same shallow shortcut, not slowly grokking their way toward carry.

So I pivoted. Between the flat seed results and a limited compute budget, chasing a lucky plain-transformer run looked like a bad bet. The more interesting question was whether a different architecture would exhibit the carry behavior more reliably — which is what took me to the looped transformer.

The looped model generalizes better — but not reliably

A 4-layer looped block does length-generalize on S₅ where the plain transformer capped out. Trained to length 24, it stays near-perfect out to ~depth 30 and degrades gracefully past that — clearly beating the ~9–12 ceiling I kept hitting with the vanilla model.

But here’s the first surprise: it only happens on about 2 of 3 seeds. Same architecture, same data, same recipe — two seeds generalize cleanly (cutoff ~25, ~0.98 accuracy); the third lands in a failing regime and never gets off the ground (~chance). This mirrors the lottery Li describes for plain transformers — the mechanism you get is partly luck of init and data order — except looping seems to raise the odds rather than remove the lottery.

The causal test: it’s re-deriving, not carrying

For the seeds that do generalize, I ran a directed counterfactual swap. The idea: at some position, overwrite the model’s internal running state with a different valid state (a class-mean translation in the residual stream — an on-manifold edit), then watch what happens downstream.

  • If the model carries state, corrupting it at position p should corrupt everything after p — the wrong state propagates forward.
  • If the model re-derives, downstream positions recompute the correct state from the raw actions and shrug off the swap.

The result was unambiguous:

  • At the swapped position: the edit takes — the model reports the counterfactual state (~99%).
  • One or more positions downstream: the model reports the correct original state ~100% of the time. The perturbation does not propagate at all.

That’s the re-derive signature — the same one the plain model shows.

I also probed along the loop-iteration axis, and it tells a consistent story: the state at a position is built up progressively across loop iterations (undecodable for the first few iterations, then snapping in around iteration 4–6). So the recurrence is doing real sequential computation — it’s just spending that computation re-deriving each position to more depth, not threading a running state across positions.

So the mechanism is something like “parallel-per-position, deepened-by-iteration.” Looping buys a higher length-generalization ceiling by giving each position more effective depth to re-derive with — not by teaching the model to carry.

Checking I didn’t fool myself

A no-propagation result is easy to get for boring reasons — swapped at the wrong spot, swapped too late, one lucky sequence — so I swept it:

  • Position: swapped at positions 2, 6, and 10 — identical re-derive signature at all three.
  • Timing: injected the perturbation both late (final loop iteration) and early (iteration 4, right as the state first becomes decodable) — still no downstream propagation either way.
  • Seeds: both generalizing seeds give near-identical numbers.
  • The failing seed never tracks well enough for the analysis to have anything to bite on — which is itself a check that the tooling isn’t manufacturing signal.

The one thing I haven’t fully pinned down: my counterfactual edit is a class-mean translation, and I’d want to confirm that operator is a faithful stand-in for the model’s real state-direction before leaning too hard on the quantitative claim. Treat the numbers as strong-but-preliminary.

Where this leaves me

Two tentative takeaways:

  1. Even an architecture built for recurrence didn’t discover the carry algorithm. It found a better re-derivation instead. If carry is what you want, looping alone may not get you there — the natural follow-up is architectures with genuine sequential state (DeltaNet / DeltaProduct-style linear-attention variants), the one family theorized to actually carry.
  2. Reliability is the sharper question. The fact that the “good” solution only shows up ~2/3 of the time — and that this seems to be a property of the optimization, not just the architecture — is the thread I find most worth pulling. What makes a training run land in the generalizing basin versus the failing one?

This is very much tinkering-in-progress, and I’m not convinced I won’t come back to it — getting reliable carry to actually occur feels like a genuinely interesting research question, not just an engineering nuisance.
